# Planning Skill Overhaul — Summary

Session goal: make the `desplega:planning` skill **clean, simple, straight to the point** and **always use sub-agents for heavy work**. Iterated via `/file-review` on `SKILL.md` and `template.md`, with a critique loop using `/radical-candor:feedback`.

## Core Changes

### Shape D — single source of truth

`SKILL.md` collapsed from a 5-step process tree to:

- **Setup** (autonomy mode, commit-per-phase preference, optional learning recall)
- **The 10 Rules** (single ordered list — no parallel Process Steps section)
- **Commit Integration** (operational notes)
- **Success Criteria Format** (one-line pointer to template)

The 10 rules absorb operational detail: rule 2 has the sub-agent menu, rule 4 has the section-by-section loop, rule 10 has the close-out sequence.

### Removed (redundant or stale)

- "When to Use" section
- "Establish preferences upfront" boilerplate
- File-review preference question (now always-on)
- `thoughts-locator` agent reference (doesn't exist in this plugin)
- "Required Structure" / "Heading Hierarchy" / "Validation Checklist" subsections (template owns the format; Haiku validator owns enforcement)
- "Key Discoveries" subsection in plans (folded into Current State Analysis)
- "Testing Strategy" + "References" sections in plan template (replaced by single `## Appendix`)

### New / Restructured

- **Rule 1 — Scaffold first**: pre-create the plan file from template before any research, so it grows incrementally and the user can correct course early.
- **Rule 2 — Sub-agent everything heavy**: file reads, research, validation. Default `run_in_background: true`.
- **New skill `desplega:ask-user`**: thin-wrapper conventions for `AskUserQuestion` (when to use, when not, formatting, good/bad examples). Other skills point at it instead of duplicating boilerplate.
- **Three Success Criteria buckets** in the plan template:
  - **Automated Verification** — runnable commands (tests, lint, build)
  - **Automated QA** — agent-driven proof of work (browser-use, screenshot diff, CLI walkthrough)
  - **Manual Verification** — only what truly needs a human (exception, not default)
- **QA Spec** is now an *external link* to a `desplega:qa`-generated doc — reserved for cross-cutting / evidence-heavy QA. Inline scenarios are gone.
- **Rule 9 — Haiku validator**: a `general-purpose` sub-agent with `model: haiku` validates plan structure before reveal.
- **Rule 10 — Hand off to fresh session**: explicit "open a new Claude Code session and run `/desplega:implement-plan <path>`" so the planner's research drops out of memory.
- **Important Guidelines additions**: concrete deliverables per phase, radical-candor pushback, propose splitting big plans.

### Template (`template.md`) tightened

- Overview now has **Motivation** + **Related** lines for quick reference
- Implementation Approach is bullet one-liners
- Phase Overview is 1–2 sentences naming the deliverable
- Single `## Appendix` replaces "Testing Strategy" and "References" — also holds follow-up plans, derail notes

### Consumer-skill sync (broke chain otherwise)

- **`desplega:phase-running`** — Step 4 handles all three Success Criteria buckets; report includes `Automated Verification`, `Automated QA`, `QA Doc: <path>` separately.
- **`desplega:implementing`** — handles `QA Doc: <path>` reports by invoking `desplega:qa` against the linked doc; ask-user adopted; file-review always-on.
- **`desplega:verifying`** — checkbox audit + Step 5 re-run cover both runnable buckets (Automated Verification + Automated QA); ask-user adopted.

### Ask-user adoption (interactive skills)

- `researching`, `reviewing`, `qa` — Working Agreement boilerplate replaced with one-line reference to `desplega:ask-user`; file-review always-on.
- `questioning` — minimal mention added (skill is intentionally ceremony-free).
- **`brainstorming` deliberately opts out** — its one-question-at-a-time Socratic discipline conflicts with ask-user's batching convention. Override is documented in its Working Agreement.

### `desplega:qa` ownership

New "Ownership: QA Doc Lives Here, Not in Plans" section explains:
- Routine per-phase QA → inline `Automated QA` bucket in the plan, run by `phase-running`.
- Cross-cutting / evidence-heavy QA → separate doc generated by this skill, linked from the plan's QA Spec block.
- Step 1 split into plan-driven / separate-spec / direct-invocation modes.

### Plugin versioning

`cc-plugin/base/.claude-plugin/plugin.json` bumped 1.13 → **2.0.0** (major — breaking changes to skill structure, template, and the SKILL.md-to-template-to-consumer contract).

## Core Learnings (Meta)

1. **We refactored on theory.** None of the changes were dogfooded against a real planning task during the session. Flagged via `/radical-candor:feedback`. Risk: real use will surface issues we didn't anticipate.

2. **Shape D collapsed structure cosmetically, not operationally.** Rule 10 has a 6-item close-out sub-list (file-review → /review → learning → QA docs → AskUserQuestion → tell user). Rule 4 has a 4-section iteration loop. The "10 instructions" framing under-counts cognitive load — operationally there are still ~20 things to do. Honest framing: 10 rules, some of which contain mini-processes.

3. **Conventions can't always be shared.** Brainstorming's one-question-at-a-time pattern conflicts with `desplega:ask-user`'s batching guidance. Resolved by explicit opt-out, but it means "uniform conventions across skills" is partial — some skills have legitimate reasons to deviate.

4. **`Automated QA` vs `QA Doc` boundary needed clarification.** Initially fuzzy. Resolved as: per-phase = inline checklist (proof of work); cross-cutting/evidence-heavy = linked doc. Default to inline; escalate to doc when warranted.

5. **Pre-creating the plan file (Rule 1) is a structural improvement.** Plan grows incrementally; user can correct course early; file isn't lost on partial failure. Worth adopting in other doc-creation workflows (research, brainstorm).

6. **File-review iteration is a real process tool.** Two passes during this session (SKILL.md, then template.md) caught 25 concrete review comments that drove the bulk of the changes. Worth invoking on any non-trivial doc.

7. **Plugin major bump (2.0) is appropriate.** Template structure changed, Success Criteria buckets changed, consumer skills changed. Anything depending on the old contract will break. Flagging this in release notes is a TODO.

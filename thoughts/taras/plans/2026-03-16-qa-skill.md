# QA Skill for cc-plugin/base — Implementation Plan

## Overview

Add a QA (functional validation) skill to `cc-plugin/base/` that captures test evidence (screenshots, recordings, links) and produces QA reports in `thoughts/*/qa/`. The skill works standalone and integrates into the existing skill chain via optional per-phase QA specs in plans and handoff additions to verify/review.

## Current State Analysis

The base plugin has 8 skills: brainstorming, researching, planning, tdd-planning, implementing, phase-running, verifying, reviewing. There is **no QA skill**. The `validate-thoughts.py` hook blocks writes to any subdirectory other than `research/`, `plans/`, and `brainstorms/`. The reviewing skill classifies documents by path but has no `/qa/` entry. The verifying skill hands off only to `/review`.

### Key Discoveries:
- `cc-plugin/base/hooks/validate-thoughts.py:29-61` — Hook blocks `qa/` subdirectory writes; needs new pattern + branch
- `cc-plugin/base/skills/reviewing/SKILL.md:63-70` — Path classification table; needs `/qa/` row
- `cc-plugin/base/skills/reviewing/SKILL.md:176-198` — Type-specific handoffs; needs QA option for plans
- `cc-plugin/base/skills/verifying/SKILL.md:142` — Hands off to `/review`; needs `/qa` option
- `cc-plugin/base/skills/implementing/SKILL.md:94` — Invokes phase-running; needs QA spec awareness
- `cc-plugin/base/skills/phase-running/SKILL.md:79-112` — Report format; needs QA spec status
- `cc-plugin/base/skills/planning/SKILL.md:153-168` — Phase writing section; needs QA Spec guidance
- `cc-plugin/base/.claude-plugin/plugin.json:4` — Version `1.5.1`; bump to `1.6.0`
- All commands follow thin-wrapper pattern: frontmatter → parse flags → invoke skill
- All skills follow consistent structure: Working Agreement → Autonomy → Process Steps → Review Integration → Handoff

## Desired End State

- A `/qa` command that invokes a new `desplega:qa` skill
- QA reports persisted at `thoughts/*/qa/YYYY-MM-DD-<topic>.md` with rich evidence (screenshots, videos, links)
- Optional `### QA Spec` sections in plan phases, executed during implementation
- Phase-running agents report QA spec status (`passed`/`pending`/`n/a`)
- Reviewing skill can review QA documents with type-specific criteria
- Verifying skill offers QA as a handoff option
- Plugin version bumped to 1.6.0

## What We're NOT Doing

- **Not building a test runner** — QA uses existing tools (qa-use, manual testing), not a new framework
- **Not modifying brainstorming or researching skills** — QA integrates only downstream from planning
- **Not creating a `/qa-use` command wrapper** — The skill uses qa-use if available, but doesn't wrap it
- **Not making QA specs mandatory in plans** — They're explicitly optional per phase
- **Not adding `qa/` directories upfront** — They get created when the first QA report is written

## Implementation Approach

Six phases, ordered by dependency: first unblock the hook and create scaffolding, then build the core skill, then integrate into the chain from planning → implementing → phase-running → reviewing → verifying, and finally bump the version.

Each phase touches a focused set of files to keep changes reviewable. Phases 3–5 (integrations) are independent of each other but all depend on Phase 2.

## Phase 2: Core QA Skill

### Overview
Create the main QA skill SKILL.md that defines the full QA process: context gathering, test case design, test execution (with qa-use or manual), evidence capture, result recording, verdict, and knowledge persistence.

### Changes Required:

#### 1. Create QA skill definition
**File**: `cc-plugin/base/skills/qa/SKILL.md` (new)
**Changes**: Create the full skill definition following the consistent pattern across all existing skills. Structure:

**Frontmatter**: `name: qa`, `description: Functional validation skill...`

**Working Agreement**: Standard pattern — AskUserQuestion primary, establish preferences upfront, autonomy modes.

**User Preferences** (upfront via AskUserQuestion):
1. **QA approach**: Manual-only / Browser automation (qa-use) / Mixed
2. **File review preference**: Use file-review for inline feedback on QA report?

**When to Use**: User invokes `/qa`, another skill references `desplega:qa`, user asks to test or validate a feature.

**Autonomy Mode**: Standard table (Autopilot/Critical/Verbose).

**Process Steps**:
1. **Context & Scope** — Determine the source of QA specs. The skill handles three input modes:
   - **Plan path provided** → Read the plan, extract `### QA Spec` sections from each phase, aggregate into QA document
   - **Separate QA spec document provided** → Read the standalone QA spec (could be a dedicated file in `thoughts/*/qa/` or any markdown with test scenarios), use it as the basis
   - **No source / PR / issue / description only** → Build test cases from scratch with the user
   Create QA document from template in all cases.
2. **Test Case Design** — Define test cases (happy path, edge cases, error scenarios). If sourced from a plan, aggregate phase QA specs. If sourced from a separate spec, use those scenarios as the starting point and augment with exploratory cases. If browser automation selected and qa-use available, design qa-use test steps.
3. **Execute Tests** — For browser automation: use qa-use CLI (explore → snapshot → interact → screenshot). For manual: guide user through steps, collect evidence. For each test case, record actual result and pass/fail.
4. **Capture Evidence** — Screenshots via qa-use or user-provided, video URLs, log output, links to external systems.
5. **Record Results** — Update QA document with actual results, evidence links, pass/fail per test case.
6. **Verdict** — Aggregate results into PASS / FAIL / BLOCKED. Summary sentence.
7. **Persist QA Knowledge** — If the project's CLAUDE.md or agents.md doesn't document how QA is done, offer to add a QA section. Also persist patterns to memory if they emerge across QA sessions.

**Browser Automation Default**: Check for qa-use availability (look for `qa-use:*` in available skills). Check if CLAUDE.md specifies a different QA tool. Fall back to manual if neither.

**Review Integration**: If file-review available and user opted in, launch file-review on completed QA report.

**Workflow Handoff**: AskUserQuestion with:
- Run post-QA verification (→ `/verify-plan`)
- Run review on QA report (→ `/review`)
- Done

# Rozum (Розум) v1 — Implementation Plan

## Overview

Ship the first shippable version of **rozum** — a Claude Code session learning system. It captures raw user prompts live (UserPromptSubmit hook), runs end-of-session analysis via background agents to infer intent and derive outcome learnings, stores everything in a single SQLite DB, and surfaces learnings back into future sessions via hooks + MCP tools. v1 targets dogfooding in the agent swarm and a backfill of historical ai-toolbox sessions.

## Current State

- Brainstorm exists at `thoughts/taras/brainstorms/2026-04-12-session-analyzer.md` (commit `3d38e51`).
- No code: `rozum/` and `cc-plugin/rozum/` do not exist yet. Greenfield.
- Repo conventions available to model on:
  - CLI template — `brain/` (Bun + TypeScript + commander + `@libsql/client`).
  - Plugin + MCP server + hooks template — `cc-plugin/teams/` (MCP tools, SQLite at `~/.desplega/desplega.db`, lifecycle hooks).
  - Hook-only template — `cc-plugin/base/` (Python hooks emitting `hookSpecificOutput`, e.g. `plan_checkbox_reminder.py`).
- Session JSONL files live at `~/.claude/projects/<encoded-cwd>/<session-uuid>.jsonl` — encoded cwd = `/` → `-`.
- Marketplace manifest at `.claude-plugin/marketplace.json` lists current plugins; no rozum entry.

## Desired End State

After v1 ships:

- `bun link`-able CLI `rozum` with commands: `init`, `doctor`, `summarize`, `analyze`, `backfill`, `list`, `stats`, `bench`, `dismiss`, `mcp`, `hook`.
- Single SQLite DB at `~/.rozum/rozum.db` with tables `sessions_analyzed`, `intents`, `learnings`, `nudges`, `schema_migrations`. All learnings carry `project_path` + `scope` (`project|global`).
- Claude Code plugin `cc-plugin/rozum/` registered in the marketplace:
  - `UserPromptSubmit` hook captures pending intent rows.
  - `Stop` hook emits `hookSpecificOutput.additionalContext` telling the main agent to spawn a bg Task that calls `mcp__rozum__*` tools to analyze the session.
  - MCP server exposes learning/intent tools for in-session agents.
  - Slash commands `/rozum:analyze`, `/rozum:list`, `/rozum:stats`, `/rozum:backfill`, `/rozum:dismiss`, and skill `rozum-expert`.
- Offline/backfill path runs analysis via `claude -p --model haiku` with JSON schema output (no live agent needed).
- Dogfood evidence: historical ai-toolbox sessions backfilled, `rozum stats` and `rozum bench` print real aggregated data, sample learnings spot-checked for quality.
- **Not shipping as an npm package on first cut.** v1 installs via `bun link` inside the repo; npm publish is deferred until the schema is proven in dogfood. `package.json` is set up for publish but the release job is manual.

## Key Design Decisions

Confirmed with Taras during planning (see `thoughts/taras/plans/2026-04-13-rozum-v1.md` history for the discussion trail):

1. **Stack:** All TypeScript + Bun — CLI, hooks, MCP server in one `rozum/` package. Hooks are shell shims that call `rozum hook <name>`; no separate bun runtime is required per hook install beyond the CLI on `PATH`.
2. **Execution model:** Hook → main CC agent → background Task. The Stop hook emits `hookSpecificOutput.additionalContext` instructing the running agent to spawn an Agent tool call that uses `mcp__rozum__*` tools. `claude -p --model haiku` is the offline backend reserved for `rozum analyze` (single session) and `rozum backfill` (many sessions).
3. **Intent triggers:** `UserPromptSubmit` captures raw user text cheaply; `Stop` fires the bg analysis. Two-phase.
4. **Storage:** Single DB at `~/.rozum/rozum.db`. Every learning row has `project_path` + `scope` (`project|global`). Cross-project aggregation is a `GROUP BY pattern_hash`. Project-scoped patterns seen in ≥2 distinct `project_path` values auto-promote `scope` to `global`.
5. **Agent-swarm integration:** v1 installs the plugin globally with no swarm-specific wiring. The swarm benefits passively. Bridging to `mcp__agent-swarm__inject-learning` is a v1.1 concern.
6. **Two abstractions, one implementation each (v1):**
   - `SourceAdapter` — how sessions are discovered and parsed per harness. v1 ships `ClaudeCodeAdapter` (JSONL at `~/.claude/projects/...`). Keeps the door open for Cursor/Windsurf/Aider adapters later without reshaping internals.
   - `Analyzer` — how a batch of work-units is turned into intents + outcome learnings. v1 ships `ClaudeAnalyzer` (shells out to `claude -p`). Keeps the door open for OpenAI/local analyzers later.
7. **Analysis is map-reduce over work-units, not one big prompt.** A "work-unit" = one user turn plus all agent/tool activity up to the next user turn (or session end). Map step: per-unit intent extraction via small LLM call. Reduce step: synthesize session-level outcome learnings from the map outputs. This avoids ever sending an oversized prompt, makes per-unit analysis parallelizable, and isolates retries to single units.
8. **Tool outputs over N bytes (default 2000) are auto-trimmed** to `{head 500}... [TRIMMED X bytes] ...{tail 500}` before entering any LLM context. Applied in the adapter layer so every analyzer benefits.
9. **The analyzer runs sandboxed.** Each `claude -p` invocation is spawned with `cwd=mktemp -d`, with the summarized + redacted session written as a file inside that tmp dir. The analyzer is invoked with `--bare` (no tools, no skills, no hooks) and structured output enforced via `--json-schema <path>`. Model is configurable (`--model`), default `claude-haiku-4-5-20251001`.
10. **Out of scope for v1** — see the dedicated [What We're NOT Doing](#what-were-not-doing) section below.

## What We're NOT Doing

Explicitly deferred or excluded from v1 to keep the scope shippable:

- **Live PostToolUse heuristic counters** — no re-read counters, error-rate thresholds, or context-consumption nudges fired mid-session. v1 only analyzes at Stop.
- **Automatic artifact generation** — no CLAUDE.md patch generation, no hook codegen, no skill scaffolding. v1 stores textual suggestions only; humans or future-Claude implement them.
- **Full benchmark harness** — `rozum bench` in v1 is a stats summary over already-stored learnings. No before/after cohort analysis, no A/B, no synthetic task replay.
- **Additional source adapters** — only `ClaudeCodeAdapter`. No Cursor, Aider, Windsurf, Devin adapters yet. The `SourceAdapter` interface exists so adding one is a contained change.
- **Additional analyzers** — only `ClaudeAnalyzer`. No OpenAI, local LLM, or rule-based analyzers.
- **Cross-project learning injection at SessionStart** — v1 does not auto-inject global learnings into new sessions. Learnings are accessed on demand via `mcp__rozum__list_learnings_for_project` or `rozum list`.
- **Agent-swarm bridge** — no push into `mcp__agent-swarm__inject-learning`. Plugin installs globally and collects passively.
- **Npm publish of the `rozum` package** — v1 ships via `bun link` locally only.
- **Web UI, dashboard, TUI** — CLI + MCP only.

### Phase 2 — Session chunker (work-units), auto-trim, Zod schemas

**Goal:** Turn a stream of `SessionTurn`s from any `SourceAdapter` into a list of bounded, model-friendly **work-units** ready for map-step analysis. Apply auto-trim to oversized tool outputs. Define the fixed Zod schemas that the analyzer contract is built on.

A **work-unit** = one user turn + every agent/tool turn up to the next user turn (or session end). Work-units are the unit of map-reduce: each is small enough to fit one Haiku call; the reduce step synthesizes across them. This replaces the brainstorm's "one big summary" approach and solves the ">7MB JSONL" problem structurally.

**Changes:**

- `rozum/src/schemas/intent.ts` — Zod schema: `{ turn_index, raw_user_text, inferred_intent, actions_summary, outcome_assessment: 'match'|'partial'|'drift'|'unknown' }`. Export JSON Schema via `zod-to-json-schema` (add dep).
- `rozum/src/schemas/learning.ts` — Zod schema matching brainstorm's `OutcomeLearning`: `{ pattern, severity: 'high'|'medium'|'low', cost_description, improvement_type: 'deterministic'|'prompting'|'mixed', suggestion, affected_scope: 'global'|'project', recurrence_potential: 'high'|'medium'|'low' }`. Export JSON Schema.
- `rozum/src/schemas/work_unit.ts` — Zod schema for `WorkUnitAnalysis`: `{ unit_index, intent: Intent, signals: string[] (0-N candidate learnings as free text), errors_observed: string[], tool_use_count: number }`. This is the per-unit map output.
- `rozum/src/session/trim.ts` — `trimLargeContent(str, maxBytes=2000)` → if over, returns `{head 500}... [TRIMMED X bytes] ...{tail 500}`. Also strips ANSI codes. Unit tests.
- `rozum/src/session/chunker.ts` — `chunkSession(turns: AsyncIterable<SessionTurn>): AsyncIterable<WorkUnit>` where `WorkUnit = { unit_index, user_turn, assistant_turns[], tool_events[], duration_s, trimmed_bytes }`. Each tool event's content is passed through `trimLargeContent` before entering the work-unit. Work-units cap at ~8k tokens estimated (cheap char count); if a single unit still blows past, we split at tool-event boundaries with a `[split 1/N]` marker and emit sub-units.
- `rozum/src/session/stats.ts` — `sessionStats(workUnits): { total_units, total_turns, tool_histogram, error_count, interrupt_count, final_state, total_trimmed_bytes }` — cheap structural stats used by summary/stats commands.
- `rozum/src/cli/summarize.ts` — new command `rozum summarize <session-id-or-path> [--source claude-code] [--json|--pretty]` prints the list of work-unit headers + aggregate stats. With `--json` emits full work-unit structure (for piping into analyzer manually).
- `rozum/test/session/chunker.test.ts` — fixture: 3-user-turn session → 3 work-units; each carries the right tool events.
- `rozum/test/session/trim.test.ts` — oversized content → truncated with markers; under-threshold → untouched.
- `rozum/test/schemas/learning.test.ts` — valid + invalid fixtures.

### Phase 3 — Analyzer interface + ClaudeAnalyzer (sandboxed, `--bare`, `--json-schema`) + map-reduce pipeline

**Goal:** Full offline analysis path via a pluggable `Analyzer` abstraction. `rozum analyze <session>` runs the map-reduce pipeline: for each work-unit call `Analyzer.mapUnit()` → collect `WorkUnitAnalysis[]` → call `Analyzer.reduceLearnings()` → dedup + store. v1 ships `ClaudeAnalyzer` which spawns `claude -p` in a sandbox tmp dir with `--bare` and `--json-schema` for enforced structured output; the model is configurable.

**Changes:**

- `rozum/src/redact/index.ts` — regex-based redactor for common secret shapes (sk-*, gh[ps]_*, AWS-style keys, JWT-ish, `password=`, `api_key=`, bearer tokens, private key PEM blocks). Replaces with `[REDACTED:<type>]`. Applied to every work-unit before it leaves the process. Unit tests.
- `rozum/src/analyzers/types.ts` — `Analyzer` interface:
  ```ts
  interface Analyzer {
    readonly name: string;
    mapUnit(unit: WorkUnit, ctx: AnalyzerCtx): Promise<WorkUnitAnalysis>;
    reduceLearnings(
      analyses: WorkUnitAnalysis[],
      sessionMeta: SessionMeta,
      ctx: AnalyzerCtx,
    ): Promise<OutcomeLearning[]>;
  }
  type AnalyzerCtx = { abortSignal: AbortSignal; debug?: boolean };
  ```
  This is what lets future analyzers (OpenAI, local, rule-based) drop in without touching the pipeline.
- `rozum/src/analyzers/registry.ts` — `getAnalyzer(name, config): Analyzer`. v1 registers only `claude`.
- `rozum/src/analyzers/claude/prompts.ts` — two prompts: `buildMapPrompt(unit)` and `buildReducePrompt(analyses, meta)`. Each prompt is small (<4k tokens typical) and references the relevant Zod schema via filename inside the sandbox (so `--json-schema` can point at it).
- `rozum/src/analyzers/claude/sandbox.ts` — `withSandbox<T>(fn: (dir: string) => Promise<T>): Promise<T>` — creates `mktemp -d` under `$TMPDIR/rozum-XXXX`, writes the relevant schema JSON files + the work-unit payload into it, invokes `fn` with the tmp dir, cleans up (even on throw).
- `rozum/src/analyzers/claude/index.ts` — `ClaudeAnalyzer` implements `Analyzer`. Each `mapUnit` / `reduceLearnings` call:
  1. Opens a sandbox dir with `withSandbox`.
  2. Writes `input.json` (the redacted work-unit or analyses array) + `schema.json` (the relevant Zod-exported JSON Schema) inside it.
  3. Spawns `claude -p --bare --model <configured> --json-schema ./schema.json --input-file ./input.json` with `cwd=sandboxDir` and stripped env (only `HOME`, `PATH`, `TMPDIR`). Uses `Bun.spawn` with `AbortSignal`.
  4. 60s timeout per call, 2 retries on 5xx/parse errors, exponential backoff.
  5. Parses stdout as JSON, validates with the Zod schema, returns typed result.
  
  Exact `claude -p` flag names (`--bare`, `--input-file`, `--json-schema`) are subject to the installed `claude` CLI version; the implementation verifies support at startup and falls back to equivalent flags (`--no-tools`, stdin piping, schema-in-prompt) with a warning if any are missing. This keeps v1 installable even on older `claude` CLIs.
- `rozum/src/analyzers/claude/config.ts` — resolves model: `--model` flag > `ROZUM_CLAUDE_MODEL` env > `~/.rozum/config.json` `analyzers.claude.model` > default `claude-haiku-4-5-20251001`.
- `rozum/src/pipeline/analyze.ts` — orchestrates map-reduce:
  ```ts
  async function analyzeSession(sessionId, opts: {
    adapter: SourceAdapter;
    analyzer: Analyzer;
    concurrency?: number;  // default 4
    dryRun?: boolean;
  }): Promise<{ intents: Intent[]; learnings: OutcomeLearning[]; stats }>;
  ```
  Uses `p-limit` for concurrent map-step calls, awaits all, then runs one reduce call, then dedup+promote.
- `rozum/src/learning/dedup.ts` — `patternHash(learning)` = SHA1 of normalized (lowercased, whitespace-collapsed) `pattern` + `scope`. On store: `INSERT ... ON CONFLICT(pattern_hash, project_path) DO UPDATE SET recurrence_count = recurrence_count + 1, last_seen = now`.
- `rozum/src/learning/promote.ts` — after any write, if a learning has `scope='project'` and its `pattern_hash` appears in ≥2 distinct `project_path` rows, promote the canonical row to `scope='global'`.
- `rozum/src/cli/analyze.ts` — `rozum analyze <session-id-or-path> [--project <path>] [--source claude-code] [--analyzer claude] [--model <model>] [--dry-run] [--concurrency N] [--debug]`. `--debug` dumps each outgoing prompt + schema to `~/.rozum/debug/<timestamp>/` so privacy can be audited. Records `sessions_analyzed` row with `source_adapter` + `analyzer_name` columns.
- `rozum/test/redact/redact.test.ts` — assert all known secret types redacted.
- `rozum/test/analyzers/claude/sandbox.test.ts` — sandbox dir created + cleaned up even on throw; cwd is the tmp dir, not the caller's cwd.
- `rozum/test/analyzers/claude/config.test.ts` — model resolution precedence.
- `rozum/test/learning/dedup.test.ts` — insert same pattern twice → count = 2.
- `rozum/test/learning/promote.test.ts` — insert project-scoped pattern in two project_paths → auto-promotes to global.
- `rozum/test/pipeline/analyze.test.ts` — pipeline with a stub `Analyzer` that returns canned `WorkUnitAnalysis` per unit; asserts map-reduce wiring, dedup, DB writes.

`sessions_analyzed` gets an additional `analyzer_name TEXT NOT NULL DEFAULT 'claude'` column (added as a migration under `schema_migrations`).

- `rozum/src/hooks/sweeper.ts` — invoked from `UserPromptSubmit` (cheap, piggybacks on existing hook): finds any `sessions_analyzed` row with `status='delegated'` older than 30 min → sets `status='stale'` and records a row in `nudges` (so `rozum stats` can surface the rate of stuck delegations). Also deletes orphaned `intents` rows (pending for >7 days with no `sessions_analyzed` entry) — lightweight GC.
- `rozum/src/cli/hook.ts` — dispatcher: `rozum hook user-prompt-submit` / `rozum hook stop` routes to the right handler.

## Rollout / Rollback

- **Rollout:**
  1. Merge to `main` after all phase commits pass.
  2. In each machine that wants rozum: `cd rozum && bun run build && bun link` → `rozum` on PATH.
  3. In Claude Code: `/plugin marketplace update desplega-ai-toolbox` → `/plugin install rozum@desplega-ai-toolbox`.
  4. Restart any active Claude Code sessions (hooks + MCP config are picked up at session start).
  5. Do not publish the `rozum` npm package on first cut.
- **Rollback (partial — plugin bug in production):**
  1. In Claude Code: `/plugin uninstall rozum@desplega-ai-toolbox`. Hooks and MCP stop firing immediately; existing sessions continue without rozum.
  2. Leave `~/.rozum/rozum.db` in place — no repo state is touched, learnings are preserved for post-mortem.
  3. Fix forward on a branch, re-release.
- **Rollback (full — abandon v1):**
  1. `/plugin uninstall rozum@desplega-ai-toolbox` everywhere it's installed.
  2. `rm -rf ~/.rozum/` (DB + debug dumps + hook.log).
  3. `bun unlink` in `rozum/` to remove the binary from PATH.
  4. `git revert <range-of-phase-commits>` on `main` (each phase is its own commit, so the revert range is well-defined).
  5. Update `.claude-plugin/marketplace.json` to drop the rozum entry (if the revert didn't already).
- **Blast-radius notes:** rozum never writes to user repos, never calls git, never modifies files outside `~/.rozum/`. Max damage from a bug is a noisy hook log, a bloated `~/.rozum/rozum.db`, or (critical-risk) a prompt injection that biases the main agent. The `--debug` dump path exists specifically so prompt-injection concerns are auditable.

## Open Questions / Known Limitations (deferred)

- Live PostToolUse heuristic counters (re-read/error-rate nudges) — defer to v1.1. v1 relies only on Stop-triggered bg analysis.
- CLAUDE.md patch / hook codegen — deferred to v2. v1 only stores suggestions.
- Real benchmark harness with before/after cohorts — v1 `bench` is a stats summary, not an A/B test.
- Cross-project learning *injection* (surfacing global learnings in SessionStart) — not wired up in v1. Adding a SessionStart hook that queries `list_learnings_for_project(scope='global')` and injects top-N is a small v1.1 PR.
- Bridge to `mcp__agent-swarm__inject-learning` — v1.1.
- Schema migrations (beyond v1 initial) — v1 ships the migration runner (`src/db/migrate.ts`) and schema v1 (`001_init.sql`) + v2 (`002_analyzer_name.sql`). Any post-v1 change is a new `NNN_*.sql` file.
- **Redaction is regex-only and therefore imperfect.** Custom secrets (internal URLs, customer PII, company-specific tokens) will NOT be caught. v1 treats this as accepted risk and documents it prominently in `rozum/README.md`. Users who care should review the `--debug` dumps before enabling the plugin on sensitive projects. Future: pluggable redactor registry and an opt-in org-specific regex file at `~/.rozum/redactors.json`.
